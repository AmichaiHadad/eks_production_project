# **Multi-Region AWS EKS Infrastructure & Application Deployment Plan**

## **1\. Infrastructure Provisioning with Terraform & Terragrunt**

**Multi-Region & VPC Design:** Deploy two identical infrastructures (one in **us-east-1** and one in **us-west-2**) for high availability and disaster recovery ([Scale applications using multi-Region Amazon EKS and Amazon Aurora Global Database: Part 1 | AWS Database Blog](https://aws.amazon.com/blogs/database/part-1-scale-applications-using-multi-region-amazon-eks-and-amazon-aurora-global-database/#:~:text=applications%20and%20let%20Kubernetes%20manage,Amazon%20EKS%20to%20learn%20more)). In each region, create a dedicated **VPC** for the EKS cluster with **multi-AZ** support. Include at least **3 private and 3 public subnets** (one pair per AZ) (DevOps eks cloud project (1).pdf). Attach an **Internet Gateway** for public subnets and a **NAT Gateway** (in each AZ’s public subnet) for outgoing internet access from private subnets (DevOps eks cloud project (1).pdf). Configure **route tables** so public subnets route externally via the IGW, and private subnets route 0.0.0.0/0 to the NAT. Use **security groups** to restrict traffic (e.g. allow node-to-node and node-to-control-plane communication, and limit public inbound access only to necessary endpoints). This multi-AZ network provides resilience against AZ failures.

* **Terraform Modules:** Utilize the latest stable **Terraform (v1.11+)** ([Releases · hashicorp/terraform · GitHub](https://github.com/hashicorp/terraform/releases#:~:text=v1)) and **Terragrunt** to define infrastructure as code. Leverage official AWS Terraform modules for VPC and EKS (DevOps eks cloud project (1).pdf) to ensure robustness. For example, use the AWS VPC module to create subnets, IGW, NAT, and route tables, and the AWS EKS module for cluster and node groups. Organize code into **reusable modules** (e.g., `vpc`, `eks-cluster`, `node_groups`) and keep region-specific variables (like AWS region, AZs, and CIDRs) in Terragrunt config files. Terragrunt will help maintain **DRY** principles by allowing common module usage while injecting region/env specific values (DevOps eks cloud project (1).pdf). Maintain a clear **repository structure** (e.g., separate folders for `networking`, `eks`, `monitoring`, `services`, `data` stacks as outlined) (DevOps eks cloud project (1).pdf) and use Terragrunt to orchestrate dependencies (e.g., VPC must apply before EKS, which before node groups).

* **Environment Isolation:** Use **Terraform state segregation** per region. Terragrunt can be configured with two directories (or overlays) for the two regions (e.g., `us-east1` and `us-west2`), each referencing the common modules with region-specific inputs. This ensures changes can be applied to one region at a time. Also, separate **workspaces or state files** for components (network, cluster, apps) to isolate changes. Version control the Terraform code, and use a **CI pipeline** or manual process to run `terragrunt plan`/`apply` in each region’s stack sequentially for initial provisioning (DevOps eks cloud project (1).pdf).

* **Latest Tool Versions:** Verify all tool versions for compatibility. For example, use **Kubernetes v1.27+** (EKS latest) to ensure compatibility with Argo CD v2.14+ and Karpenter v1.x. Use **Helm v3** (latest stable) for chart deployments. Confirm the AWS provider plugin for Terraform is up-to-date (AWS provider \~v5.x) to support new resources, and Terragrunt is the latest stable (v0.x). Keeping tools updated ensures support for new features and security patches across the deployment.

## **2\. EKS Cluster Configuration (Control Plane)**

**Cluster Creation:** Deploy an **Amazon EKS** cluster in each region using Terraform. Make the cluster name and Kubernetes version configurable (via variables) (DevOps eks cloud project (1).pdf). The EKS control plane is managed by AWS, but specify configuration to harden it:

* Enable **private endpoint access** for the Kubernetes API and **disable public access**, so the cluster API is reachable only within the VPC ([Control network access to cluster API server endpoint \- Amazon EKS](https://docs.aws.amazon.com/eks/latest/userguide/cluster-endpoint.html#:~:text=You%20can%20enable%20private%20access,access%20to%20the%20API%20server)) ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=,communicate%20with%20the%20Kubernetes%20API)). This involves setting `endpointPrivateAccess = true` and `endpointPublicAccess = false` in the EKS module. Using a private API endpoint is a security best practice to avoid exposure of the cluster API on the internet ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=,communicate%20with%20the%20Kubernetes%20API)). If administrative access is needed from a workstation, use a VPN or bastion host inside the VPC or AWS Systems Manager Session Manager to run `kubectl` commands. Optionally, restrict the API server to specific CIDR ranges or a security group if some limited public access is required for admins (DevOps eks cloud project (1).pdf).

* **IAM Roles for EKS:** Create an IAM role for the EKS control plane (passed in Terraform via `role_arn` for the cluster). Attach the AWS managed policies **AmazonEKSClusterPolicy** and **AmazonEKSServicePolicy** to allow the control plane to manage AWS resources like networking and load balancers. This role is required for the cluster to create ELBs for Services, etc. Additionally, set up IAM **OIDC identity provider** for the cluster (Terraform `aws_eks_identity_provider_config`) to later enable IRSA for service accounts (used by Argo CD, Karpenter, etc.).

* **Cluster Add-ons:** Use latest EKS platform versions. Enable core logging for audit, authenticator, scheduler, controller-manager logs to CloudWatch (via Terraform `enabled_cluster_log_types`). Install the **VPC CNI plugin** (usually default in EKS) and consider enabling **Kubernetes Dashboard** only internally if needed (with proper authentication). No SSO is used, so rely on IAM authentication (aws-auth config map) and Kubernetes RBAC for access control. Also, plan for cluster scaling: while Karpenter will handle node scaling, you might disable the legacy Cluster AutoScaler to avoid conflict (we will rely on Karpenter’s dynamic provisioning).

* **Multi-Cluster Consideration:** Each region’s cluster is independent. To manage deployments across them, you can either deploy separate Argo CD instances (one per cluster) or use a single Argo CD to manage multiple clusters via its contexts. In this plan, we will set up each cluster with its own Argo CD instance for simplicity and autonomy. For global services (like the domain `blizzard.co.il`), DNS (Route 53\) could be configured with latency-based or failover routing to the ALB in each region. This ensures users are directed to the closest or backup region. (This DNS setup is external to EKS but important for multi-region HA).

## **3\. Node Groups & Compute Configuration**

Provision four **Managed Node Groups** in each cluster, each dedicated to a specific workload type (DevOps eks cloud project (1).pdf) (DevOps eks cloud project (1).pdf). Using multiple node groups enables workload isolation and optimal sizing per workload ([Using Multiple Node Groups on Amazon EKS | by Hareesh Iyer | Medium](https://medium.com/@iyer.hareesh/using-multiple-node-groups-on-eks-57ba1ecd1068#:~:text=Workload%20Segregation)). All node groups use the latest EKS-optimized AMI (Amazon Linux 2 or AWS Bottlerocket) for the Kubernetes version.

* **General Node Group Settings:** Each node group spans across the three private subnets (multi-AZ) for high availability. Enable **auto-scaling** on all groups with appropriate min/max/desired counts (DevOps eks cloud project (1).pdf). For example, start with 1–2 nodes minimum in each group and allow scaling up to a higher limit (e.g., 3-5 nodes) to handle load and failures. Use instance types sized to the workload: smaller instance type for lightweight workloads, larger for heavy ones. All nodes use **On-Demand** instances for critical stability, but non-critical groups (like monitoring) could use a mix of On-Demand and Spot for cost efficiency (configurable through the Terraform EKS module, e.g., specifying a Spot node group). Ensure each node group is diversified across instance types if using Spot to avoid spot capacity issues ([Using Multiple Node Groups on Amazon EKS | by Hareesh Iyer | Medium](https://medium.com/@iyer.hareesh/using-multiple-node-groups-on-eks-57ba1ecd1068#:~:text=The%20best%20practice%20is%20to,into%20multiple%20Spot%20capacity%20pools)).

* **Node Group Breakdown:**

  * **Monitoring Node Group:** Purpose – run monitoring pods (Prometheus, Grafana, exporters). Use moderate instance types (e.g., t3.large) to handle metrics workloads. Size for memory-intensive usage (Prometheus can use significant memory). Taint this group with `monitoring=true:NoSchedule` (for example) and label it accordingly (e.g., `node-role=monitoring`). This ensures only monitoring pods (with a toleration) run here ([Using Multiple Node Groups on Amazon EKS | by Hareesh Iyer | Medium](https://medium.com/@iyer.hareesh/using-multiple-node-groups-on-eks-57ba1ecd1068#:~:text=Workload%20Segregation)).

  * **Management Node Group:** Purpose – run management and control-plane add-ons like **Argo CD** (and possibly AWS Load Balancer Controller or other operators). Use small to medium instances (e.g., t3.medium) as Argo CD and similar tools are lightweight. Label as `node-role=management` and taint if needed (e.g., `management=true:NoSchedule`) so that only Argo CD and critical control pods land here.

  * **Services Node Group:** Purpose – run the **application services** (the Python web app pods). Use instance types optimized for web workload (e.g., m5.large for balanced CPU/memory). This group will host the majority of user-facing pods. It can be kept untainted (so it’s the default scheduling target for general apps) or use a toleration strategy where app pods are explicitly scheduled here. Ensure it can scale out on demand (especially since KEDA/Karpenter will add pods/nodes for this workload).

  * **Data Node Group:** Purpose – run **stateful services** like MySQL and Elasticsearch (DevOps eks cloud project (1).pdf). Use **storage-optimized** or memory-heavy instances (e.g., r5.xlarge or i3 for IO) that are capable of handling databases. These should be on **EBS-backed storage** for persistence. Taint these nodes (e.g., `role=data:NoSchedule`) and add tolerations to MySQL/Elasticsearch pods so that only these stateful sets run on data nodes. This isolation prevents other workloads from consuming resources needed for databases and ensures that if these nodes must be optimized or configured (e.g., with certain kernel settings), they are separate.

* **IAM Instance Roles:** Create a distinct IAM role for each node group (or at least separate roles for the **data** group for additional EBS permissions). Attach the standard EKS worker node policies: **AmazonEKSWorkerNodePolicy, AmazonEKS\_CNI\_Policy, AmazonEC2ContainerRegistryReadOnly** and **CloudWatchAgentServerPolicy** (if sending logs to CloudWatch). These allow nodes to join the cluster, use the CNI, pull images, and (optionally) push logs. The data nodes’ role should also include permissions to manage EBS volumes (though typically the worker node policy suffices for attaching EBS to pods). By using distinct roles, we could later restrict certain AWS permissions to specific workloads if needed (for example, only the data node role could be allowed to interact with EBS, etc.). All roles are mapped in the EKS aws-auth configmap so nodes can join. Also enable **IRSA (IAM Roles for Service Accounts)** on the cluster OIDC provider ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=access%20docs.aws.amazon.com%20.%20,and%20AWS%20CloudTrail%20to%20mitigate)) ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=,54%20AWS%20IAM%20Access)) – this will be critical for Karpenter, external DNS, and other controllers needing AWS API access without using node credentials.

* **Security Groups for Nodes:** EKS creates a cluster SG by default. Configure each node group to use the cluster’s security group and an additional per-group security group if needed. For instance, allow **intra-cluster traffic** (nodes to nodes, nodes to control plane) on required ports: Kubernetes API (if control plane endpoint is internal and accessed via cluster SG), etcd ports (managed by AWS), and application ports as needed. The data node group might need an SG rule to allow traffic from app nodes on MySQL (3306) and Elasticsearch (9200) ports, but ideally, within cluster, all pod-to-pod communication is via cluster networking (so handle via NetworkPolicy at the pod level for zero-trust). Define Kubernetes **NetworkPolicies** to restrict namespace communications (e.g., only allow the app namespace to talk to the MySQL service, etc.), adding an extra security layer inside the cluster.

## **4\. Continuous Delivery Setup with Argo CD and GitOps**

**Deploy Argo CD:** Install **Argo CD** (latest stable v2.14+ in 2025\) on the **Management** node group of each cluster to enable GitOps continuous delivery (DevOps eks cloud project (1).pdf). Use Terraform’s Helm provider or a Terragrunt module to deploy the official Argo CD Helm chart (ensuring it’s v2.x compatible with our K8s version). Alternatively, apply the Argo CD manifest YAMLs.

* **Argo CD Namespace & Access:** Create a dedicated namespace (e.g., `argocd`). Deploy Argo CD components (server, repo-server, application-controller, dex (if any), redis, etc.). Expose the Argo CD API server with a Service of type LoadBalancer for easy access. For security, use an internal LoadBalancer (only accessible within the VPC) or restrict the inbound SG to trusted IPs if external. Argo CD can also be accessed via port-forward or CLI if not exposing a LoadBalancer (DevOps eks cloud project (1).pdf). Since we are not using SSO, we will rely on Argo CD’s local user accounts. **Retrieve the initial admin password** from the `argocd-initial-admin-secret` in the Argo CD namespace (DevOps eks cloud project (1).pdf) and immediately change it to a strong password. Document this admin password securely (or better yet, integrate with an external secret store). Disable the `argocd-initial-admin-secret` after use (Argo CD by default suggests deleting this secret once the admin password is set ([Getting Started \- Argo CD \- Declarative GitOps CD for Kubernetes](https://argo-cd.readthedocs.io/en/stable/getting_started/#:~:text=cd,purpose%20than%20to%20store))). Configure RBAC within Argo CD for any additional users or read-only access as needed (for instance, create a deployer role if multiple team members will use the UI).

* **Git Repository (GitOps) Setup:** Create a Git repository (e.g., on GitHub) to host all application and deployment manifests (DevOps eks cloud project (1).pdf). Organize it as follows:

  * `/app` directory for the Python application source code.

  * `/helm-chart` directory for the Kubernetes Helm chart of the application.

  * `/argocd` directory for Argo CD configurations (like Application or ApplicationSet manifests).

  * Optionally, separate directories for each microservice or environment if needed in the future.  
     Push this repository to a source control platform (GitHub), and ensure Argo CD has access. If the repo is private, generate a read-only deploy key or personal access token and configure it in Argo CD’s secret (`argocd-secret`) for repo access. Since our pipeline will also push to this repo, ensure the CI has commit access.

* **Argo CD ApplicationSet:** Implement Argo CD in a **declarative GitOps** manner. Define an **ApplicationSet** YAML in the `argocd/` folder that uses the **Git directory generator** or **list generator** to manage all our deployments (DevOps eks cloud project (1).pdf). For example, one ApplicationSet can have multiple Applications: one for the app, one for MySQL, one for Elasticsearch, one for Prometheus, etc., each pointing to respective Helm charts in the repo. Alternatively, create individual Argo CD `Application` manifests for each component. Enable **auto-sync** and **self-heal** in Argo CD so that it will automatically apply changes and correct drift (DevOps eks cloud project (1).pdf). This means whenever we push a change to the Git repo (new application version or config), Argo CD will detect it and apply to the cluster. This GitOps approach ensures the cluster’s state is always reconciling towards the Git declarative state.

* **HTTPS Access (ACM Integration):** Request or import an **AWS ACM Certificate** for `*.blizzard.co.il` in each region (or in a shared region like us-east-1 if using Route53 with us-east-1 for ACM). Tag the certificate for use. Deploy the **AWS Load Balancer Controller** (ALB Ingress Controller) in the cluster (it can be installed via Helm chart in the management node group) to manage ingress resources ([Serve distinct domains with TLS powered by ACM on Amazon EKS | Containers](https://aws.amazon.com/blogs/containers/serve-distinct-domains-with-tls-powered-by-acm-on-amazon-eks/#:~:text=AWS%20Elastic%20Load%20Balancers%20provide,new%20ALB%20for%20each%20ingress)) ([Serve distinct domains with TLS powered by ACM on Amazon EKS | Containers](https://aws.amazon.com/blogs/containers/serve-distinct-domains-with-tls-powered-by-acm-on-amazon-eks/#:~:text=Controller%2C%20formerly%20called%20the%20AWS,new%20ALB%20for%20each%20ingress)). The controller needs IAM permissions (attach the **AWSLoadBalancerControllerIAMPolicy** to an IRSA service account). Once deployed, Argo CD itself can optionally be exposed via an Ingress with host `argocd.blizzard.co.il` if external access is needed. For the Argo CD server, using a self-signed cert is acceptable for admin use, but for consistency, we could also use ACM: create an Ingress with annotation `alb.ingress.kubernetes.io/certificate-arn` pointing to the ACM cert ARN ([Serve distinct domains with TLS powered by ACM on Amazon EKS | Containers](https://aws.amazon.com/blogs/containers/serve-distinct-domains-with-tls-powered-by-acm-on-amazon-eks/#:~:text=target%20subnets%20for%20the%20ALB%2C,powerful%20mechanism%20to%20provision%20the)) and host `argocd.blizzard.co.il`. The ALB Controller will provision an ALB with HTTPS listener using that certificate. This provides a secure GUI endpoint for Argo CD.

## **5\. CI/CD Pipeline and Container Image Management**

**CI Pipeline (GitHub Actions):** Set up a GitHub Actions workflow to automate building and deploying the Python application. When code is pushed to the main branch of the **app** directory, the pipeline should trigger:

1. **Build and Push Docker Image:** Use Docker to build the Python app image. The Dockerfile will containerize the app (using Python base image, copying app code, installing dependencies). Tag the image as `<ECR_repository>:<git-sha>` or `<version>` (DevOps eks cloud project (1).pdf). Authenticate to **Amazon ECR** (create an ECR repo for the app) and push the image. Using ECR in each region: push to both regions’ ECR registries if we want each cluster to pull from its local region for speed and redundancy. (Alternatively, push to a public registry or a single region, but cross-region ECR pull could add latency).

2. **Update Helm Chart & GitOps Repo:** After pushing the image, update the Helm chart’s values.yaml with the new image tag (and image repository if using region-specific ECR URLs). This can be done in the workflow by using sed or a Helm values updater. Commit and push this change to the Git repository (in the `helm-chart/` or a version manifest). Since Argo CD is watching the repo, this commit will trigger Argo to deploy the new image. This GitOps deployment avoids direct kubectl applies. Alternatively, use **Argo CD Image Updater** tool, but for simplicity, updating values in Git is straightforward.

3. **Chart Versioning:** Optionally, update the Helm chart version and app version in a Chart.yaml if maintaining versioned charts. This isn’t strictly necessary with Argo tracking Git HEAD, but it’s good practice for artifact integrity.

4. **Terraform Changes Pipeline:** Although not explicitly requested, consider a separate pipeline for infrastructure. If Terraform code changes (infrastructure updates), use GitHub Actions to run Terragrunt with plan/apply. This could be manual (requiring approval) as infrastructure changes are less frequent and more sensitive.

**Artifact Security:** Implement best practices in the CI: scan the Docker image for vulnerabilities (e.g., using Trivy or AWS ECR scan on push) ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=,and%20replace%20deployed%20images%20where)). Ensure no secrets are in the code repo – rely on external config for secrets (see Secrets Management below). Use Git branching strategy (dev/staging/prod branches) if promoting changes through environments, and Argo CD can be pointed at specific branches or tags for different clusters/environments.

## **6\. Python Application Deployment (Helm & Kubernetes)**

**Application Helm Chart:** Develop a **Helm chart** for the Python web application (DevOps eks cloud project (1).pdf). This chart will define Kubernetes manifests for:

* A **Deployment** running the Python app container. Configure it with replicas (start with 2 pods for HA). Set container resources (requests/limits) appropriate for the app (e.g., 100m CPU, 200Mi memory requests as baseline). Use environment variables or ConfigMap for the welcome message text, and the Tel-Aviv weather API key (if needed) and endpoint. Mount a downward API field to get the pod name or container ID (the app can read its own hostname or pod name to display container ID).

* A **Service** exposing the Deployment (ClusterIP). The service will be targeted by an Ingress or load balancer for external traffic. Use port 5000 as the container port (DevOps eks cloud project (1).pdf) and map it to Service port 80 (or 5000\) as needed.

* An **Ingress** resource to expose the app at `https://app.blizzard.co.il`. Annotate it for the AWS Load Balancer Controller (e.g., `kubernetes.io/ingress.class: alb`). Include the ACM certificate ARN annotation for TLS ([Serve distinct domains with TLS powered by ACM on Amazon EKS | Containers](https://aws.amazon.com/blogs/containers/serve-distinct-domains-with-tls-powered-by-acm-on-amazon-eks/#:~:text=target%20subnets%20for%20the%20ALB%2C,powerful%20mechanism%20to%20provision%20the)). Path route all `/*` traffic to the app Service. Use **external-dns** (if set up) or manually create a DNS CNAME in Route 53 pointing `app.blizzard.co.il` to the ALB’s DNS name. The Ingress will ensure traffic is encrypted and reaches the app pods.

* **Kubernetes Secrets:** If the app needs secrets (like an API key for weather data, or DB credentials for MySQL), do not hard-code them. Instead, create a Secret resource via Helm (values for which are provided securely, e.g., via CI injecting them or using an External Secrets Operator). For example, define a secret `weather-api-key` and mount it as env var. **Secrets management strategy** is detailed later (see section 10).

* **ConfigMap:** Include a ConfigMap for non-sensitive configs (e.g., a default welcome message or configuration toggles). The app should read the welcome message and other config from env or configMap to avoid code changes for simple text updates.

**Dynamic Content Implementation:** The Python app (assumed to be a Flask or FastAPI app) will fetch and display: client IP, a welcome message, container identifier, and current Tel-Aviv temperature (DevOps eks cloud project (1).pdf). Achieve this by:

* Reading the client IP from request headers (or inferring via Flask `request.remote_addr`).

* Storing a welcome message in an environment variable (populated via ConfigMap).

* Getting the container ID or pod name from an environment variable (Kubernetes exposes pod name via `metadata.name` env if configured, or simply use the hostname since K8s sets hostname \= pod name by default).

* For temperature: use a weather API (like OpenWeatherMap) called by the app. The API key and endpoint URL should be provided via Secret/ConfigMap. Because the app runs in Kubernetes, ensure outbound internet access is available (the nodes in private subnets will egress via NAT Gateway to call the API). Alternatively, deploy a simple service or use a pre-fetched data for temperature if no external API is allowed.

**Deploy via Argo CD:** Once the Helm chart and values are ready and pushed to Git, Argo CD (through the ApplicationSet) will deploy the application to the **Services** node group. It will create the Deployment, Service, Ingress, etc. **Verify** in Argo CD’s UI that the Application shows healthy and synchronized. After deployment, test the app URL: access `https://app.blizzard.co.il`. You should see the page with your IP, the welcome message, container/pod info, and the Tel-Aviv temperature. Because we used ACM, the browser will trust the certificate (ensure the domain’s DNS is pointed correctly). If using self-signed cert as a fallback (not recommended in prod), you’d have to accept the certificate warning (DevOps eks cloud project (1).pdf), but in our plan we prefer ACM for real TLS.

* **Ingress and Load Balancer:** The AWS Load Balancer Controller will have created an ALB. Check that it attached the ACM certificate and that the target group has the EC2 instances (nodes) registered. Since our Service is ClusterIP, the controller actually creates a target group with IP targets (the pod IPs) or via NodePort depending on configuration. We prefer IP mode for ALB target so it targets pods directly. Confirm the security group on the ALB allows inbound 443 from the internet and that the nodes’ SG allows the ALB to communicate on the node port if needed (the controller usually manages this).

* **Scaling Config for App:** Initially, we set replicas to 2\. We will later integrate **KEDA** to auto-scale pods based on metrics, but even so, configure the HorizontalPodAutoscaler (if not using KEDA for that metric) or at least set an upper limit of replicas to avoid runaway scaling. KEDA will create an HPA under the hood, so ensure metrics server is functioning (KEDA installs one). For now, manual HPA could be set to scale on CPU as backup (e.g., target 50% CPU, scale 2-10 pods).

## **7\. Deploying Stateful Services (MySQL & Elasticsearch)**

Stateful services are deployed on the **Data** node group to isolate their resource usage and attach persistent storage.

* **MySQL Deployment (Helm & Argo CD):** Use a **Helm chart** for MySQL (for example, Bitnami’s MySQL chart, which is well-maintained). Configure it as a **StatefulSet** with 1 replica (or more if HA is needed) (DevOps eks cloud project (1).pdf). Key configurations in `values.yaml`:

  * Set the **storage class** to AWS EBS (e.g., `gp3`) and request a persistent volume (e.g., 20Gi or as needed) (DevOps eks cloud project (1).pdf). The chart will create a PersistentVolumeClaim; EKS will provision an EBS volume in the same AZ as the pod via the default StorageClass. Because the node group spans AZs, consider using a StorageClass that allows multi-AZ volume (EBS is AZ-bound, so the StatefulSet pods will be constrained to a single AZ each; spreading MySQL replicas across AZs means each has its own volume in that AZ).

  * Provide MySQL credentials via values (username, password, database name) (DevOps eks cloud project (1).pdf). Store these in Kubernetes Secrets (the Helm chart usually auto-creates a secret for the root password). Use secure random passwords (the pipeline or Terraform can generate and supply these as values securely).

  * Enable MySQL’s service (ClusterIP for in-cluster access). No public exposure.

  * Node scheduling: add a **nodeSelector** or toleration in the values so that MySQL pod schedules to the `Data` node (e.g., nodeSelector `node-role=data`). Also add the toleration for the `data:NoSchedule` taint we set on data nodes. This guarantees MySQL runs on the intended nodes.

  * If high availability is needed: consider running MySQL in a Primary/Replica mode with 2+ replicas. The Bitnami chart supports a primary with multiple read replicas. That would require additional configuration (and the app would need to differentiate read vs write). For now, a single instance or a simple primary-replica is sufficient.

* Deploy MySQL via Argo CD by creating an Application (or as part of an ApplicationSet) pointing to the MySQL chart. Once synced, verify the MySQL pod is running on a data node and an EBS volume is attached to that node (check `kubectl get pvc` and AWS console for volume). Verify you can connect to MySQL internally (use a Kubernetes Job or port-forward to run a mysql client). **Data persistence**: if the pod restarts or moves (within same AZ), data on EBS persists. In case of AZ failure, manual intervention may be needed to deploy a new instance in another AZ with backup/restore, since EBS cannot move AZs (for cross-AZ resilience, one could use AWS Aurora or a multi-AZ MySQL, but that’s outside scope).

* **Elasticsearch Deployment:** Use the official **Elasticsearch Helm chart** (from Elastic or Bitnami’s **Elasticsearch** chart) to deploy a cluster as a StatefulSet (DevOps eks cloud project (1).pdf). Configuration:

  * **Cluster size:** For basic HA, run 3 master/data nodes (each pod a master-eligible data node). This will tolerate one node failure. Set replicas to 3 and allow the chart to create 3 PVCs. Use EBS volumes (e.g., 20–50Gi each) for each Elasticsearch node’s storage (DevOps eks cloud project (1).pdf). Each pod will get an EBS in its AZ.

  * Configure **node roles** in values (all can be data+master for simplicity, or dedicate one as master-only if needed) (DevOps eks cloud project (1).pdf).

  * Set resource limits: Elasticsearch needs memory (e.g., 2Gi heap per pod at least). Ensure the Data node group instance type can accommodate that (e.g., r5.large has \~15Gi, allocate \~4Gi to ES, rest for OS).

  * **Tolerations and Node Affinity:** Similar to MySQL, configure tolerations for `data:NoSchedule` and nodeSelector to ensure all ES pods run on data nodes. Spread the 3 pods across AZs (StatefulSet will automatically spread by scheduling each ordinal to a different AZ if the node group has instances in each AZ and if using default volume provisioning – since each PVC will bind in the AZ of the node it lands on, it’s tricky but generally it tries to spread if possible).

  * Expose Elasticsearch with a ClusterIP Service for internal access. No public ingress (we are using it internally for logs and possibly search by the app). If needed, one can deploy Kibana later with an ingress for log viewing, but Grafana can also query Elasticsearch if configured.

* Deploy via Argo CD (Application manifest pointing to ES chart). Verify pods running and forming a cluster (check ES cluster health via its API or logs – it should be green). Ensure each ES pod is on a different node (kubectl get pods \-o wide to see node names). Also verify PVCs and EBS volumes for each. Set up an index retention and shard allocation as needed (perhaps via an Index Lifecycle Management policy, though not mandatory for this plan).

* **Integrate Application with MySQL:** Update the Python application to use MySQL:

  * Add a MySQL client library to the app (e.g., PyMySQL or SQLAlchemy). On each HTTP request (or at least periodically), have the app connect to MySQL and record request metadata (could log the timestamp and client IP, for example) (DevOps eks cloud project (1).pdf). The database connection info (host, user, password, db name) should come from environment variables or a Kubernetes Secret (mounted as env) (DevOps eks cloud project (1).pdf). The MySQL service DNS would be `mysql.<namespace>.svc.cluster.local` (if deployed in same namespace as app, just use service name). Ensure the app can reach it: since app runs on services nodes and MySQL on data nodes, but both are within cluster networking, this is fine. If NetworkPolicies are in place, allow app namespace to connect to MySQL’s namespace on port 3306\.

  * Test this integration: after deployment, generate a request to the app and then use a MySQL client to see if the record was inserted into MySQL. This confirms connectivity and that credentials are correct. Use **Kubernetes Secret** for DB password to avoid putting it in Git; the Secret can be created by Helm (values containing the password, which could be set via pipeline or Terragrunt with a generated random secret).

* **Ship Application Logs to Elasticsearch:** Implement log forwarding so that all app logs are sent to Elasticsearch for analysis (DevOps eks cloud project (1).pdf). Use the **EFK stack** approach:

  * Deploy a **Fluentd** or **Fluent Bit** DaemonSet on the cluster (on all nodes or at least on services nodes where app runs). Fluentd will tail container logs from `/var/log/containers/*app*.log`. Use a Fluentd configuration to parse the log (if structured JSON, even better) and send to Elasticsearch. The Elasticsearch endpoint is the internal service we set (e.g., `elasticsearch.master.svc:9200`). Provide credentials if ES has auth (the Elastic chart can set up basic auth; for simplicity, maybe disable auth or use an internal user). Alternatively, deploy **Elastic’s Filebeat** as a DaemonSet to ship logs.

  * If using Fluentd: use the **fluentd-elasticsearch** Kubernetes addon (often there’s a pre-made image and config in EKS add-ons or community charts). This will create an index in Elasticsearch (like `kubernetes_logs` or similar) and store container logs. Confirm that the app’s logs (including perhaps the request logs or any errors) appear in Elasticsearch by querying the index (e.g., `_search` via curl or using Kibana/Dev Tools).

  * **MySQL and ES logs:** We can also deploy Fluentd on data nodes to ship MySQL logs and Elasticsearch logs themselves into Elasticsearch. However, shipping ES’s own logs into itself can be noisy; it's often fine to focus on app logs. The MySQL chart might not emit to stdout (since it’s a container, it will log to stdout/stderr); Fluentd will capture those as well by default.

* **Verify Logging:** Once Fluentd is running, generate some application traffic. Then query Elasticsearch’s index (via Kibana or curl) to see those logs. We expect to see the Python app logs (which could include the request details, etc.). This completes the EFK integration, giving us centralized log analysis. (Grafana can even be configured with the Elasticsearch data source to visualize logs, or we could deploy Kibana via Helm for a dedicated UI.)

* **Persistent Data & Backup:** For MySQL, set up a basic backup strategy. At minimum, take periodic snapshots of the EBS volume (can be automated with AWS Data Lifecycle Manager) or use a Kubernetes CronJob with `mysqldump` to S3. Similarly, for Elasticsearch indices, snapshots to S3 could be configured using Elastic’s snapshot API (requires an S3 repository plugin and IAM permissions on the ES pods – which we can supply via IRSA if needed). This is an advanced step but important for disaster recovery in multi-region: you would restore data from one region to another if one fails. In our scenario, you could periodically replicate data to the other region’s cluster (e.g., MySQL binlog replication cross-region, or Elasticsearch cross-cluster replication), but those are complex; snapshots suffice for now.

## **8\. Monitoring and Alerting (Prometheus, Grafana, Alertmanager)**

Set up a comprehensive monitoring stack on the **Monitoring** node group, covering cluster health, application performance, and database metrics.

* **Prometheus Deployment:** Use the **official Prometheus Helm chart** (from Prometheus community). This can deploy a Prometheus server and required exporters. Configure it to:

  * **Scrape Kubernetes metrics:** Enable **kube-state-metrics** and **node-exporter**. The kube-state-metrics service will be deployed (Helm can install it as a sub-chart or we deploy separately) to generate cluster state metrics (deployments, pods, etc.), and node-exporter as a DaemonSet on all nodes to collect node-level metrics (CPU, memory, disk) (DevOps eks cloud project (1).pdf). Ensure the Prom scrape configs include these.

  * **Scrape application metrics:** If the Python app exposes custom metrics (e.g., via a `/metrics` endpoint in Prometheus format), add a scrape job for it (or annotate the app deployment for Prometheus to autodiscover via service monitors). If not, at least monitor the app via Kubernetes metrics (like CPU, memory usage, and maybe request rate if using an NGINX ingress controller – but with ALB, we can’t get pod request metrics from ALB easily, so consider adding a Prometheus metric in the app to count requests).

  * **MySQL Exporter:** Deploy a **mysqld\_exporter** sidecar or separate Deployment to scrape MySQL metrics (like queries per second, connections, buffer pool stats). The MySQL Helm chart can optionally deploy a sidecar exporter – enable that in values if available (DevOps eks cloud project (1).pdf). If not, deploy the **prometheus-mysql-exporter** with a connection to the MySQL service (provide it the DB credentials via Secret). Prometheus will scrape this exporter on a given port, yielding metrics on MySQL (e.g., `mysql_up`, query count, etc.).

  * **Elasticsearch Exporter:** Similarly, deploy an **elasticsearch\_exporter** (there is a Prometheus exporter for ES) (DevOps eks cloud project (1).pdf). This can run as a Deployment on the monitoring nodes, configured to connect to the Elasticsearch service and poll cluster health, index metrics (docs count), node stats, etc. Provide it credentials if ES has auth. Prometheus scrapes it on its listen port.

  * **Alertmanager:** The Prometheus Helm chart can also install **Alertmanager** or you use the separate **kube-prometheus-stack** chart which includes it. We will deploy Alertmanager (one replica) and configure it to receive alerts from Prometheus. Set up a ConfigMap for Alertmanager with routing and receivers for notifications (DevOps eks cloud project (1).pdf).

  * **Prometheus configuration:** In values, define `additionalScrapeConfigs` to include MySQL and ES exporter targets if not using ServiceMonitors. Also, include **pod monitor** for KEDA (KEDA metrics server) and **cAdvisor** (though cAdvisor (container metrics) are usually scraped via the kubelet or via node-exporter these days). Ensure Prometheus has adequate resource (e.g., 2 CPU, 4Gi memory) and persistent storage (optional; if we want to retain metrics on disk beyond memory limits, use an EBS PVC of 20Gi+).

* Install Prometheus via Argo CD (another Application). After deployment, check Prometheus UI (you can port-forward to the service) to ensure targets are up: Kubernetes components, node-exporters, mysql-exporter, elasticsearch-exporter, etc., should all show as **UP** and scraping without errors (DevOps eks cloud project (1).pdf). Verify you can query basic metrics: e.g., `up{job="kubernetes-nodes"}` for node health, `mysql_global_status_connections` for MySQL, etc.

* **Grafana Deployment:** Deploy **Grafana** via its official Helm chart. Configure Grafana to use Prometheus as the default data source (the chart can create a data source config if given the Prometheus service URL) (DevOps eks cloud project (1).pdf) (DevOps eks cloud project (1).pdf). Use a Service of type LoadBalancer or Ingress for Grafana if you want external access. Given this is an internal tool, you might expose it internally and use port-forward or VPN for access. However, providing an HTTPS ingress (e.g., `grafana.blizzard.co.il`) with ACM certificate is a nice touch. If so, add another ingress resource with host and TLS annotation as done before. Restrict access with admin credentials:

  * Set the admin user and password via Helm values (or better, use an existing secret). Use a strong password and consider integrating Grafana with Slack or others for auth (or at least enable OTP for admin, etc.). Without SSO, we rely on Grafana’s local auth; ensure to change the default password.

  * **Dashboards:** Include or create dashboards for our stack: the Helm chart can provision dashboards via ConfigMaps or JSON definitions. Use community dashboards as starting points:

    * **Kubernetes cluster dashboard:** showing node usage, pod counts, etc. (there’s a popular one by Grafana).

    * **Prometheus / system dashboard:** if using kube-prom-stack, it ships some. Otherwise, import premade dashboards for node-exporter and kube-state-metrics.

    * **MySQL dashboard:** e.g., the “MySQL Overview” dashboard (community) to visualize queries, connections, InnoDB buffer metrics (the MySQL exporter provides these) (DevOps eks cloud project (1).pdf).

    * **Elasticsearch dashboard:** showing cluster health, indexing rate, search latency, JVM heap, etc. (There are community dashboards for ES exporter metrics).

    * **Application dashboard:** a custom dashboard to show our app metrics. If the app doesn’t expose Prometheus metrics, we rely on infrastructure metrics: use ALB CloudWatch metrics (not captured by Prometheus by default) or if we had NGINX we’d have request metrics. We can approximate with CPU/memory and any custom metrics we added (or use KEDA metrics which we’ll set up). Additionally, we can display logs in Grafana by adding Elasticsearch as a data source (Grafana can query ES for logs if needed, though using Kibana is typical for log deep-dive).

* After deploying, log into Grafana (initial credentials from Helm values). Verify the Prometheus data source is working. Import or create the dashboards. Ensure data is visible (e.g., you can see node graphs, MySQL stats, etc.). Grafana should be configured to persist its state (you might give it a PVC for storing its SQLite DB of configs, or rely on redeploy from code). We can store our dashboard JSON in Git and have the Helm chart pick them up as ConfigMaps to be provisioned – making the monitoring as code.

* **Alerting Setup (Alertmanager):** Configure **Prometheus Alerting rules** for critical conditions (DevOps eks cloud project (1).pdf). Create a ConfigMap or use the Helm chart’s `serverFiles.alerting_rules.yml` to add rules such as:

  * **High MySQL latency** or too many connections (indicator of DB overload).

  * **Elasticsearch cluster status** goes red or nodes down.

  * **High error rate or latency** on the web application (if we have such metrics; alternatively, an alert if the app pod restarts too often or if CPU is saturated).

  * **High Pod usage**: e.g., if pods are consistently at max replicas or pending due to insufficient nodes (which should trigger Karpenter).

  * **Node down** or high node CPU (to foresee need to scale up).

  * **Kubernetes events** like CrashLoopBackoff detection (could be done via Prom or via Kubernetes event exporter).

* The Prometheus Operator (if we used kube-prom-stack) would allow writing rules in YAML CRDs (PrometheusRule), but with plain Helm chart, we can embed rules. For example, an alert rule for Elasticsearch cluster health: if `elasticsearch_cluster_health_status{cluster="mycluster"} == 2` (red status) for 5 minutes, trigger a critical alert. Or for app: if Pod restart count \> 3 in 5m, alert.

   **Alertmanager**: Set up receivers:

  * **Slack Notifications:** Configure a Slack receiver with a webhook URL (obtainable by creating a Slack app or incoming webhook in your Slack workspace) ([Step-by-step guide to setting up Prometheus Alertmanager with ...](https://grafana.com/blog/2020/02/25/step-by-step-guide-to-setting-up-prometheus-alertmanager-with-slack-pagerduty-and-gmail/#:~:text=,Manage%20apps)) ([Setting Up Prometheus Alertmanager with Slack Integration](https://devopsvoyager.hashnode.dev/setting-up-prometheus-alertmanager-with-slack-integration#:~:text=Integration%20devopsvoyager,Sign)). For example, route all `severity=warning` or `critical` alerts to a Slack channel. In Alertmanager config (YAML), under `receivers`, add Slack with the webhook and channel, and under `routes` specify which alerts go there (could be all). Test by using the Alertmanager’s `/api/v1/alerts` to fire a test alert or temporarily create a condition that's easy to trigger (like make a fake metric).

  * **Email/PagerDuty:** For completeness, also set up an email receiver (using AWS SES or a company SMTP) for critical alerts, or a PagerDuty integration if available. The task specifically mentioned Slack as primary, but best practice is to have multiple channels (Slack for dev attention, PagerDuty for on-call, etc.). Ensure sensitive info (like API keys for PagerDuty or Slack) are stored in Kubernetes as secrets and referenced in the Alertmanager config via environment or volume mounts, not plaintext in Git.

  * **Verify Alerts:** Intentionally simulate an alert (e.g., scale MySQL down or change an alert threshold temporarily) to see if Alertmanager sends a Slack message. Also check that Argo CD’s Sync failures or Kubernetes events are not silently ignored – we might integrate Argo CD notifications separately if needed (Argo CD has its own notification system which can also send to Slack on deployment issues, but that’s optional).

* **Monitoring the Monitors:** Ensure **Prometheus and Grafana** themselves are healthy. Grafana can have an alert for when Prometheus is down (no data points). Also, configure **Alertmanager High Availability** if needed (in bigger setups, two replicas with a mesh to avoid single point failure, but one is fine for now given our scale).

* **Resource considerations:** The Monitoring node group should have enough capacity (if Prom and ES on same cluster, they can be heavy). If needed, adjust auto-scaling for this group or offload ES to its own cluster in future. With our setup, continuous monitoring data (Prom metrics) and logs (ES) will grow – consider retention settings (Prom default retention maybe 15 days in memory, which might be fine; ES retention for logs maybe delete after 7 days if not needed longer, to save space).

## **9\. Autoscaling with Karpenter and KEDA**

Implement a two-tier autoscaling: **KEDA** for pods (based on workload metrics) and **Karpenter** for nodes (to provide capacity when new pods need to run). This ensures the application can handle variable load automatically (DevOps eks cloud project (1).pdf) (DevOps eks cloud project (1).pdf).

* **Karpenter Installation:** Install **Karpenter v1.0+** on each cluster ([Announcing Karpenter 1.0 \- AWS](https://aws.amazon.com/about-aws/whats-new/2024/08/karpenter-1-0#:~:text=Posted%20on%3A%20Aug%2014%2C%202024)). Use the official Helm chart (Karpenter is now stable, v1.x) to deploy the Karpenter controller to the cluster. Karpenter will require:

  * **IAM Role for Karpenter Controller:** Create an IAM role with policies to allow provisioning EC2 instances. This includes permissions like `ec2:RunInstances`, `ec2:TerminateInstances`, `ec2:Describe*`, and also permissions to create/edit ASG or LaunchTemplates if needed (Karpenter can create instances directly without ASG). AWS provides a predefined IAM policy for Karpenter which can be used ([terraform-aws-modules/eks/aws | karpenter Submodule](https://registry.terraform.io/modules/terraform-aws-modules/eks/aws/19.4.2/submodules/karpenter#:~:text=An%20IAM%20instance%20profile%20for,the%20EKS%20Managed%20Node)). Associate this IAM role with a Kubernetes ServiceAccount (`karpenter-controller`) via IRSA ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=access%20docs.aws.amazon.com%20.%20,54%20AWS%20IAM%20Access)). This way, Karpenter controller running in cluster can assume the role to manage EC2.

  * **Karpenter CRDs:** The Helm chart will install CRDs like **Provisioner** or the newer **NodePool** (Karpenter v1.5+ introduces NodePool to replace Provisioner in future). After installation, define a **Provisioner** custom resource. This is the heart of Karpenter’s config: specify constraints for dynamic nodes. For example:

    * Set `clusterName = <eks cluster name>` and `clusterEndpoint = <endpoint>` if required (some setups need this, though Karpenter can auto-detect in EKS).

    * AWS-specific settings: which subnet IDs to use for new instances (use our private subnets IDs), which security groups to attach (use the cluster’s SG or a dedicated one for Karpenter nodes).

    * **Instance Types:** list allowed instance families (e.g., allow m5.large, m5.xlarge, c5.large, etc., or even say `aws.amazon.com/instance-family: [m, c, r]` with size constraints). You can also let Karpenter decide, but best to restrict to what your application can run on (e.g., exclude very small or too large).

    * **AZs:** allow all three AZs in the region to maximize spot capacity options and spread.

    * **Capacity Type:** you can allow both On-Demand and Spot in the provisioner. For example, `requirement: "capacity-type" = spot` could let it use spot for cheaper cost for ephemeral scaling. Possibly maintain a percentage or separate provisioner for spot vs on-demand if needed.

    * **Taints/Labels:** if we want Karpenter-launched nodes to only serve certain pods, we could assign taints. However, in our case, Karpenter will primarily scale the **Services** (application) workload, which is not tainted (or we can specifically target a provisioner for app pods). We might create a provisioner specifically for app pods that have, say, a label `karpenter.sh/discovery=app` or use the default and rely on KEDA to create unschedulable pods.

    * **Limits:** set a max node count or resource cap to prevent runaway scaling (budget control). e.g., max 10 nodes or max $cost.

  * Karpenter will watch for unschedulable pods. To integrate with our existing managed node groups: if an incoming workload exceeds managed nodes capacity, pods will stay pending, and Karpenter can then create new nodes to accommodate them ([Karpenter \- Amazon EKS](https://docs.aws.amazon.com/eks/latest/best-practices/karpenter.html#:~:text=,schedule%20due%20to%20resource%20constraints)). This complements managed ASGs which have their own scaling (though we set ASG max, they might not scale fast enough for sudden loads; Karpenter will step in for quicker scaling). Over time, Karpenter can also **consolidate** unused nodes (terminate them) to save cost when load drops.

* **KEDA Installation:** Deploy **KEDA v2.16+** (latest) using its Helm chart. KEDA is lightweight – it will deploy a Metrics Adapter and controller. Configure KEDA by creating a **ScaledObject** for our Python application Deployment (DevOps eks cloud project (1).pdf) (DevOps eks cloud project (1).pdf):

  * The `ScaledObject` links to the Deployment (via `scaleTargetRef`). We specify the trigger as **Prometheus** type (DevOps eks cloud project (1).pdf). For example, use the Prometheus metrics adapter to scale on HTTP request rate or latency. If our app exported a Prometheus metric (say `http_requests_total`), we could use that. Alternatively, we could use a generic metric like CPU as a trigger (though HPA does that natively). Better, we set up a custom metric: For instance, if we use the ALB’s CloudWatch metrics, we could import them via Prometheus (through CloudWatch exporter) – but that's complex. Simpler: incorporate a **Prometheus Pushgateway** or have the app push a metric of current request load. To keep it straightforward, use **CPU utilization** via KEDA (KEDA can also scale on Kubernetes metrics server metrics by using the built-in HPA compatibility).

  * **Prometheus trigger example:** KEDA can query Prometheus directly. Provide the Prometheus server URL (internal service URL) and a PromQL query and threshold ([Prometheus | KEDA](https://keda.sh/docs/1.4/scalers/prometheus/#:~:text=triggers%3A%20,a%20vector%2Fscalar%20single%20element%20response)). For instance: `sum(rate(http_requests_total{app="myapp"}[5m]))` with threshold X requests per second. If above X, scale out. Or use average latency metric if available. As another approach, use **Kubernetes HPA** on CPU which KEDA can override if needed, but we want event-driven. So we'll do: trigger type Prometheus, metric e.g. `http_requests_per_second` (if we create such a recording rule in Prom).

  * Configure `minReplicaCount` (maybe 2\) and `maxReplicaCount` (maybe 10\) in the ScaledObject (DevOps eks cloud project (1).pdf). This ensures even at idle we keep 2 pods (for HA), and at most 10 pods to avoid excessive scaling. KEDA’s controller will monitor Prometheus periodically. When the query result exceeds the threshold, it will instruct to increase replicas (essentially it creates an HPA object behind the scenes with the custom metric).

  * Deploy the ScaledObject manifest via Argo CD (so it’s in Git). Verify KEDA created an HPA (`kubectl get hpa`) for the app. You should see the metric value and target. Ensure Prometheus server is reachable by KEDA (might need to create a Kubernetes Secret with the server URL if TLS, but since Prom is cluster-internal HTTP, it’s fine). KEDA’s logs will show if it can scrape the metric. (No authentication needed if Prom is unsecured inside cluster, which it is by default) ([Prometheus | KEDA](https://keda.sh/docs/1.4/scalers/prometheus/#:~:text=The%20,vmselect%3E%3A8481%2Fselect%2F0%2Fprometheus)).

* **Autoscaling Test:** Simulate high load to verify autoscaling (DevOps eks cloud project (1).pdf). For example, run a load test (hey, JMeter, etc.) from a client to `app.blizzard.co.il`. Observe in Prometheus the increase in request rate (or CPU). KEDA should detect the metric crossing threshold and scale the Deployment (adding replicas). This new demand will cause new pods to schedule. If the current nodes (managed “Services” group) are full, these pods will be Pending. Karpenter will notice Pending pods (via the scheduler events) and will provision new EC2 instances to accommodate them ([Karpenter \- Amazon EKS](https://docs.aws.amazon.com/eks/latest/best-practices/karpenter.html#:~:text=,schedule%20due%20to%20resource%20constraints)). Within a short time, new nodes join the cluster, and pods move to Running state. Verify in AWS console that new EC2 instances (with tags identifying Karpenter) launched in the appropriate subnets. Also verify the app scaled up (e.g., now running 6 pods). The ALB ingress will automatically include the new pods in its target group (since it targets all pods via endpoints). So traffic should continue seamlessly, now spread across more pods and nodes.

* **Karpenter Tuning:** After the test, when load subsides, KEDA will scale pods down (back to min 2). Those extra pods terminate, leaving some nodes underutilized. Karpenter’s **consolidation** feature will detect nodes with no scheduled pods and terminate them after a cooldown. Ensure the Provisioner is configured with `consolidation: enabled` or similar setting to allow auto scale-down of empty nodes. You can adjust `ttlSecondsAfterEmpty` (e.g., 30-60 seconds) so that nodes are freed fairly quickly when idle, to save cost (DevOps eks cloud project (1).pdf). Also set `ttlSecondsUntilExpired` for maximum lifetime if desired (sometimes used to recycle nodes periodically).

* **Alternative/Additional Autoscaling:** We use Karpenter for node scaling instead of the classic Cluster Autoscaler. If we had chosen to not use managed node groups at all, Karpenter could manage all nodes. In our case, managed node groups handle baseline capacity and Karpenter handles burst. This hybrid is fine. If we wanted to use only Karpenter, we’d create provisioners for each type of workload with labels/taints accordingly. But using both is acceptable and gives us more control for core components (which are on managed nodes) versus autoscaled components.

* **Scaling Metrics in Grafana:** Add panels to Grafana to visualize scaling events – for example, number of pods over time (from K8s metrics), number of nodes over time (from node-exporter or cloud metrics). Prometheus will have metrics like `kube_node_status_capacity` and such, or we can rely on CloudWatch for exact scale events. But we can see things like `karpenter_nodes_created_total` metric if provided by Karpenter (Karpenter emits some metrics). Also track KEDA metrics – KEDA as a metrics server might expose metrics usage.

* **Failure handling:** Ensure that if Karpenter fails to launch nodes (e.g., AWS capacity issue), alerts are sent (Prometheus can alert on Pending pods \> N for \> T minutes). Similarly, if KEDA doesn’t trigger for some reason and pods max out, that should alarm. We want to catch if our autoscalers are not functioning.

## **10\. Security, Secrets Management, and Best Practices**

**Authentication & Access Control:** In absence of SSO, we enforce strict access via IAM and Kubernetes RBAC:

* **Cluster API Access:** Only allow authenticated IAM entities (e.g., admins’ IAM user/role) to assume the EKS admin role. Map IAM roles to Kubernetes `system:masters` in aws-auth config for admin operations. Use separate IAM roles for developers with limited access if needed, mapped to specific RBAC roles (e.g., view-only). With the API endpoint private, require users to be on VPN or a bastion to use `kubectl`. This mitigates risk of unauthorized access.

* **Argo CD Access:** Use Argo CD’s local user accounts with strong passwords. Disable the default admin after setting up alternative accounts if multiple users. Implement Argo CD RBAC so that only certain people can sync applications or change configurations. Since no SSO, consider using Argo CD's integrated Dex with static user/password or linking to an LDAP if available (if not, at least unique accounts per admin).

* **Grafana Access:** Set admin password, and create limited viewer accounts for developers as needed. Consider enabling Grafana’s built-in auth with GitHub OAuth or similar even if cluster doesn’t use SSO, Grafana can still have social login (optional). If not, at least ensure it’s not publicly reachable without VPN.

**Secrets Management:** Do not store plain secrets in Git. Use one of these strategies for best practice ([Making sense of secrets management on Amazon EKS for regulated institutions | AWS Security Blog](https://aws.amazon.com/blogs/security/making-sense-of-secrets-management-on-amazon-eks-for-regulated-institutions/#:~:text=Applications%20often%20require%20sensitive%20information,135%20GoDaddy%20Kubernetes%20External)) ([Making sense of secrets management on Amazon EKS for regulated institutions | AWS Security Blog](https://aws.amazon.com/blogs/security/making-sense-of-secrets-management-on-amazon-eks-for-regulated-institutions/#:~:text=External%20Secrets%20Operator%20,Azure%20Key%20Vault%2C%20IBM%20Cloud)):

* **External Secrets Operator (ESO):** Install the Kubernetes External Secrets Operator ([Making sense of secrets management on Amazon EKS for regulated institutions | AWS Security Blog](https://aws.amazon.com/blogs/security/making-sense-of-secrets-management-on-amazon-eks-for-regulated-institutions/#:~:text=External%20Secrets%20Operator%20,Azure%20Key%20Vault%2C%20IBM%20Cloud)) to integrate AWS Secrets Manager with Kubernetes. For example, store the MySQL password, the Slack webhook URL, and any API keys in AWS Secrets Manager. Then create ExternalSecret CRs in Kubernetes that pull those values into Kubernetes Secrets. This way, the actual secret values are managed in AWS Secrets Manager (which supports rotation and is secure), and Kubernetes only gets the synced value. This approach is recommended by AWS for managing sensitive config on EKS ([How to use AWS Secrets Manager and ABAC for enhanced secrets ...](https://aws.amazon.com/blogs/security/how-to-use-aws-secrets-manager-and-abac-for-enhanced-secrets-management-in-amazon-eks/#:~:text=How%20to%20use%20AWS%20Secrets,Manager%20using%20External%20Secrets%20Operator)) ([How do you manage secrets in Kubernetes? \- Reddit](https://www.reddit.com/r/kubernetes/comments/1bbx0ua/how_do_you_manage_secrets_in_kubernetes/#:~:text=Currently%2C%20I%20work%20a%20lot,tools%20like%20Vault%2C%20AWS)). We would need to give ESO’s service account permission to read the secrets (via IAM policy). Alternatively, use AWS SSM Parameter Store similarly.

* **Kubernetes Secrets Encryption:** Enable EKS envelope encryption with KMS for secrets at rest in etcd (a checkbox in EKS cluster creation or via Terraform). This ensures if etcd is compromised, secrets are encrypted. Even with that, avoid putting high-sensitive data in Git. If External Secrets is too complex to introduce, at least use **Helm secrets/SOPS** to encrypt values in Git and decrypt in pipeline or ArgoCD. For example, use Mozilla SOPS with a KMS key to encrypt the Secret YAMLs in the repo, and Argo CD can be configured to decrypt with that key.

* In our deployment, use Kubernetes Secrets for simplicity but populate them via Terraform or CI pipeline. For instance, Terragrunt can generate a random password for MySQL and create a Secret manifest as part of the helm values (or we manually create secret then reference in Helm values as existingSecret). Keep those random values in a secure place (like as Terragrunt output or in a password manager).

* **Application Secrets:** The Python app’s weather API key should also be in a Secret. We can use Kubernetes External Secret mapping if we installed ESO, or just manually create it and refer in Helm. Similarly, database credentials used by the app come from the Secret created by the MySQL chart or a custom secret; mount them via env vars in the Deployment.

**Update and Patch Management:** Ensure all components are pinned to specific **versions** and update regularly:

* **Kubernetes Version:** Plan cluster upgrades (EKS releases minor versions frequently). Use Terraform to manage upgrades when needed, and test in one region before the other. Argo CD and other tools compatibility must be checked (Argo CD 2.14 supports up to K8s 1.27+ likely ([Argo CD | endoflife.date](https://endoflife.date/argo-cd#:~:text=Argo%20CD%20,9))).

* **OS Patching:** Managed node groups use AWS’s AMI – enable automated patching or use a strategy to recycle nodes (EKS can do rolling upgrades of node groups). Karpenter nodes should use a specific AMI ID if you want control (you can set a custom AMI in the Provisioner). The plan should freeze on known good AMIs and update them periodically ([Karpenter \- Amazon EKS](https://docs.aws.amazon.com/eks/latest/best-practices/karpenter.html#:~:text=Lock%20down%20AMIs%20in%20production,clusters)).

* **Container Image Security:** As noted, use ECR image scanning or integrate **Amazon Inspector** to scan running images for vulns. Also, sign container images (using Docker Content Trust or Cosign) and verify in cluster if possible.

* **Least Privilege IAM:** The various IAM roles (for nodes, for Karpenter, for external-dns if used, etc.) should have only needed permissions. For instance, if using ExternalDNS to manage Route53 records for the domain, create an IAM policy scoped to just the hosted zone. Karpenter’s IAM as provided by AWS is already least-privilege for provisioning. Argo CD itself doesn’t need AWS permissions unless we store secrets in S3/git in private etc (not in this case). If any pod needs AWS API access (e.g., the weather API could be AWS API – not here though), use IRSA.

**Notifications & Logging:** Beyond Slack alerts, aggregate system logs:

* **AWS CloudWatch**: While we have our logs in Elasticsearch, it’s also a good practice to send cluster logs to CloudWatch (especially EKS control plane logs and Karpenter logs). EKS control plane logs (API server, authenticator, etc.) can be enabled to CloudWatch Logs for audit purposes. Set up a CloudWatch Log Group for each cluster’s control plane. For node logs, we could install CloudWatch agent or FluentBit to copy `/var/log/messages` if needed. However, since app logs and such are in ES, we have covered that.

* **Slack Integrations:** Consider integrating Argo CD notifications to Slack (Argo CD has a notifications component that can send deployment successes/failures to Slack). This gives visibility when new releases roll out or if something syncs out-of-sync.

* **Runbooks:** Document incident response steps (not part of deployment, but part of ops). For example, if Alert “Elasticsearch red” comes, what to do (maybe run `kubectl logs` on ES pods to see issue, etc.). Having those along with alerts helps on-call engineers.

**Testing & Validation:** After deploying all components:

* Conduct end-to-end tests: simulate region failure (e.g., kill us-east-1 cluster nodes) and ensure you can bring up the app in us-west-2 (if using DNS failover, flip it). This tests DR. If active-active, test load sharing.

* Perform load testing to ensure autoscaling works and the system stabilizes without manual intervention. Monitor that response latencies meet expectations with scaled out pods.

* Test backup/restore: e.g., take a snapshot of MySQL, restore it in another cluster or verify you can. Similarly, test that if you delete a pod with data (MySQL), on reschedule it can reattach volume and continue.

**Compliance & Best Practices:** We followed AWS best practices by isolating workloads on node groups ([Using Multiple Node Groups on Amazon EKS | by Hareesh Iyer | Medium](https://medium.com/@iyer.hareesh/using-multiple-node-groups-on-eks-57ba1ecd1068#:~:text=Workload%20Segregation)), using private API endpoints ([EKS Best Practices Guide for Security | Caylent](https://caylent.com/blog/eks-best-practices-guide-for-security#:~:text=,communicate%20with%20the%20Kubernetes%20API)), least-privilege IAM, and securing communications with TLS. We also used infrastructure-as-code (Terraform/Terragrunt) for reproducibility and GitOps (Argo CD) for continuous delivery, ensuring the deployment is **declarative and auditable**. All these measures contribute to a robust, secure, and scalable multi-region EKS deployment.

Finally, maintain thorough **documentation** of this deployment plan and configs (which this document serves as). Going forward, any changes to infrastructure or apps should go through code reviews via Git, and the plan should be updated accordingly. This ensures the multi-region Kubernetes platform remains well-managed and ready to serve production workloads reliably.
